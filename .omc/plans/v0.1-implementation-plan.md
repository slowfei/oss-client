## RALPLAN-DR Summary

### Principles (5)

1. **Frozen surface, escape hatches.** The 11 entities, 14 error codes, and 13 capabilities are immovable in v1.x. Vendor-specific richness escapes via `As(target any) bool`, `Error.Cause`, `Error.SecondaryID`, and `DriverConfig any`. No driver may add a new top-level `Code` or `Capability` value at runtime.
2. **Multi-module dependency direction is one-way.** `pkg/uos` MUST NOT import any `providers/<name>` and MUST NOT import any third-party cloud SDK. `pkg/testkit/contract` MAY import `pkg/uos` + `testcontainers-go` only. Each provider imports `pkg/uos` and exactly one vendor SDK. No `internal/` package crosses a module boundary.
3. **Stream-first, capability-first, config-explicit.** Public API treats `io.Reader`/`io.Writer` as primary; bool fields like "supports X" are illegal â€” use `capability.Report`. No global state, no init-time side effects in `pkg/uos`; every behavior is settable via explicit `Config`.
4. **Contract suite is the abstraction's contract.** `pkg/testkit/contract.RunSuite(t, factory)` enumerates the frozen `Code` Ă— `Capability` matrix. If a behavior isn't asserted by `RunSuite`, it isn't part of v1's promise. The suite must compile and assert case shape in M1 even though no driver yet exercises it.
5. **Additive-only public API in v1.x.** Any breaking change to `pkg/uos` requires a major bump. Any new `Code` or new `Capability` requires â‰Ą 2 independent providers needing the same semantic AND a minor bump on `pkg/uos` (or `pkg/uos/capability`). No exceptions for "convenience."

### Decision Drivers (top 3)

1. **M2 (AWS + MinIO) must be unblocked the day v0.1 ships.** Anything that makes M2 wait â€” ambiguous interface, missing helper type, undefined error mapping convention, missing testkit entry point â€” is a v0.1 bug. M2 is the abstraction-validation milestone; if M2 forces a redesign, v0.1 was wrong.
2. **Frozen surface erosion is the highest cost defect.** A wrong `Code` or wrong `Capability` enumerated in v0.1 forces either (a) a major bump within months of release or (b) silent provider hacks. Both destroy the abstraction's promise. Therefore v0.1 picks the 14 + 13 set defined in Â§7 of architecture_plan and writes a `surface_test.go` that pins the literal set.
3. **Multi-module developer ergonomics.** A multi-module Go layout makes `go test ./...` from the root surprising for newcomers. `go.work` mitigates this, but only if it's bootstrapped at M1 with the future `providers/<name>` modules pre-listed (or a documented procedure for adding them). Friction here delays every later milestone by hours per contributor per session.

### Viable Options

#### Option A â€” "Skeleton + interfaces only" (pure types, no runtime)

- **What it is**: define every public type, interface, constant, and request/response struct. `Registry`, `Factory.Open`, `transfer.Manager`, etc. are interface-only or panic-with-`ErrUnsupported` stubs. `pkg/testkit/contract.RunSuite` exists, asserts case-shape via reflection over the frozen sets, but performs no I/O.
- **Pros**: smallest possible v0.1; easiest to review for surface correctness; fastest to ship; least risk of locking in a wrong implementation choice.
- **Cons**: M2 starts from "no working glue" â€” first AWS driver author has to implement `transfer.Manager` orchestration logic from scratch and we discover its abstraction defects in M2 instead of M1. Risk: `transfer.Manager` design only validates against one driver before being declared frozen.
- **When this wins**: when team velocity matters more than discovering abstraction defects before M2.

#### Option B â€” "Skeleton + working defaults" (recommended)

- **What it is**: Option A plus (a) a default in-process `Registry` implementation, (b) a working `transfer.Manager` skeleton that orchestrates a `MultipartService` with a memory `StateStore` and the unknown-size policy state machine, (c) a working `httpx.HTTPConfig.NewClient()` returning a configured `*http.Client`, (d) a default `credential.StaticProvider`, (e) a working `middleware` chain that no-ops by default but composes correctly. No vendor SDK code, no provider implementation, no network I/O outside the testkit's MinIO container helper.
- **Pros**: M2 driver author's job is to write the driver only â€” orchestration, retry policy translation surface, and credential refresh framework are already there and exercised by unit tests + the testkit reflection over frozen sets. Catches abstraction defects in `transfer.Manager` before any driver is written. Establishes the vocabulary for "what is a driver concern vs. a `pkg/uos` concern."
- **Cons**: more code to review for v0.1; risk that the `transfer.Manager` skeleton bakes in a bias toward S3-style multipart that needs to be unwound at M4 (GCS resumable uploads) or M5 (Qiniu Upload Token).
- **When this wins**: when M2 throughput and abstraction-defect-discovery latency matter most. Also wins when `transfer.Manager` is acknowledged as the highest-bus-factor part of the abstraction.

#### Option C â€” "Skeleton + working defaults + reference fake driver"

- **What it is**: Option B plus a `providers/fake` (in-memory) driver inside the **root module** (NOT a separate module â€” kept inside root so it can live under `internal/fake` without violating the multi-module rule). The contract suite runs end-to-end against `internal/fake` in CI as part of the root module's tests, providing a non-trivial executable validation of the abstraction.
- **Pros**: strongest v0.1 confidence â€” `RunSuite` actually runs and goes green at M1 tag. The fake driver also doubles as a permanent unit-test fixture for transfer/middleware behavior in v1.x.
- **Cons**: an in-memory fake is so structurally different from real cloud (no network, no eventual consistency, no real auth) that "fake-passes" is weakly correlated with "real provider passes." Risk: false confidence. Also adds ~300-500 LOC to v0.1 for a fake whose long-term value is unclear once `providers/minio` ships in M2.

**Recommendation: Option B.** It maximizes M2 unblock without paying the false-confidence cost of a fake driver, and it surfaces `transfer.Manager` design defects against unit tests in M1 rather than against a real driver in M2 (which is the highest-value time to surface them). Option C's fake driver is pushed to M6 if benchmarks need it. Option A is rejected as not actually faster: the orchestration code has to be written anyway by M2, and writing it in M1 with no driver pressure produces a cleaner abstraction than writing it under M2 deadline pressure.

**No "only one viable option remained" condition**: A and B are both shippable. The recommendation is a velocity/risk tradeoff, not a forced selection.

## Pre-mortem (deliberate mode)

### Scenario 1: "v0.1 froze the wrong `transfer.Manager` boundary, M4 GCS forces a major bump within 8 weeks of release."
- **Likelihood**: Medium. GCS resumable uploads have a session URL handed back from the server, not a per-part upload-id + part-number model. If `transfer.StateStore` only persists `(uploadID, partNumber, etag)`, GCS doesn't fit cleanly.
- **Blast radius**: HIGH. A `pkg/uos/transfer` major bump cascades to every shipped provider (M2, M3) and breaks every consumer of the API. We'd be at `pkg/uos/v2.0.0` before any ecosystem formed.
- **Mitigation**:
  - `StateStore` interface uses `Save(ctx, key, data []byte) / Load(ctx, key) ([]byte, error) / Delete` â€” opaque blob, NOT a typed `(uploadID, parts)` struct (already in architecture_plan Â§4.11).
  - Driver owns serialization of its own resume state. The `transfer.Manager` is a *coordinator*, not a *schema owner*.
  - In M1 `surface_test.go`, assert that `StateStore.Save` takes `[]byte` (not a typed struct) â€” if anyone later "improves" it to a typed struct we catch it at the surface test.
  - Add an explicit `StateStore` test case in the contract suite from M1 day one: "save â†’ load â†’ equality of bytes" â€” independent of any driver.
  - Document in Â§6 of this plan that `transfer.Manager` may grow alternative *strategies* (multipart-style, session-style for GCS, FORM-style for Upyun) but the public `transfer.Manager` interface stays stable.

### Scenario 2: "Frozen surface erodes via well-meaning PR adding `ErrConcurrentModification` for Azure ETag conflict, then `CapEventNotifications` because 'Azure has it'. Within 6 months the 14 + 13 sets are 18 + 17 and the contract suite no longer pins anything."
- **Likelihood**: HIGH. This is the single most common abstraction-decay failure mode for SDKs that try to span vendors. Every vendor has "one more capability" that sounds reasonable in isolation.
- **Blast radius**: CRITICAL. The whole reason for frozen sets is so business code can write `errors.Is(err, &uos.Error{Code: uos.ErrNotFound})` and `if rep.Items[capability.CapVersioning].Availability == capability.Supported` and have those work for the lifetime of v1. If the sets grow, business code is forced to keep up, and the abstraction's value evaporates.
- **Mitigation**:
  - **Surface-pinning test**: `pkg/uos/surface_test.go` enumerates the literal 14 codes and 13 capabilities. Any addition or removal fails the test loudly with a message pointing to architecture_plan Â§7. Test runs on every CI PR.
  - **CODEOWNERS-style governance** (out of scope for v0.1 code, but documented in Â§6 of this plan): any PR that modifies `pkg/uos/error.go` constants or `pkg/uos/capability/capability.go` constants requires explicit reviewer approval citing the â‰Ą2-provider justification rule.
  - **CHANGELOG discipline**: every minor bump of `pkg/uos` documents the new code/capability, the 2 providers needing it, and links to the relevant matrix update.
  - **`Error.Code` is a `string` typed alias**, NOT an integer iota. This makes JSON serialization stable and prevents reordering accidents from being silent breaking changes.

### Scenario 3: "Multi-module + `go.work` + `pkg/testkit/contract` triangle creates a cyclic dev-loop where contributors can't run tests without Docker, the testkit refuses to compile against future module paths, and the whole repo is unfriendly to OSS contributors."
- **Likelihood**: Medium. `go.work` is well-supported in Go â‰Ą 1.22, but contributors on Windows / no-Docker often hit `testcontainers` failures and bounce.
- **Blast radius**: MEDIUM. Doesn't break the released SDK but slows velocity by 2-3Ă— and erodes the contributor base. Multi-module is also notoriously hard to refactor backward to single-module if we change our minds.
- **Mitigation**:
  - `go.work` committed at M1 with root module enumerated. Add a `scripts/add-provider-module.sh` (or documented `Makefile` target) that scaffolds the new `providers/<name>/go.mod` and adds it to `go.work`. Documented in this plan Â§1 P2.
  - **Build tag `//go:build docker`** on `pkg/testkit/contract/minio.go` so the testkit compiles cleanly without Docker present. `RunSuite` works without Docker for the cases that don't need it; the MinIO-backed cases skip with a clear `t.Skip("requires docker / testcontainers")` message.
  - `make test-no-docker` documented in `AGENTS.md` (Appendix A item) for contributors without Docker.
  - The CI matrix includes one `no-docker` job that runs unit tests + `go vet ./...` to catch build-tag regressions.
  - Module path placeholder `github.com/<owner>/object-storage-client` is **substituted via `go mod edit -module`** at first init, never via `sed` across files. All in-repo cross-module imports resolve via `go.work`'s replace-equivalent semantics, so the placeholder works in dev even before the canonical owner is set.

## 6. Migration / change-control rules for the public API

This section is the single most important deliverable for "what changes are allowed post-v0.1."

### 6.1 Versioning rules for `pkg/uos`

- **Major bump** (`v1.0.0` â†’ `v2.0.0`): any breaking change to a function signature in `pkg/uos`, `pkg/uos/capability`, `pkg/uos/credential`, `pkg/uos/transfer`, `pkg/uos/middleware`, or `pkg/uos/httpx`. Includes: removing a method from any interface; changing a method's parameter or return types; removing an exported type or constant; renaming an exported identifier; changing the underlying type of an exported alias; changing the meaning of an existing constant value.
- **Minor bump** (`v1.0.0` â†’ `v1.1.0`): any additive change. Includes: adding a new method to a non-`Client`/`BucketService`/`ObjectService`/`MultipartService`/`Signer` interface (those five are append-only requires major); adding a new exported function; adding a new exported type; adding a new field to a struct (additive only â€” never reordering or repurposing); adding a new `Code` (gated by Â§6.2); adding a new `Capability` (gated by Â§6.2). The five "frozen interface" interfaces require a major bump for any method addition because every driver implements them and an addition would break every driver.
- **Patch bump** (`v1.0.0` â†’ `v1.0.1`): bug fixes, doc updates, internal refactors with zero observable surface change, dependency updates with zero observable surface change. Patch must not change observable behavior â€” if it does, it's a minor bump.

Each subpackage tags independently from a versioning standpoint within the same module: `pkg/uos/v1.X.Y`. The subpackage `pkg/uos/capability`, `pkg/uos/credential`, `pkg/uos/transfer`, `pkg/uos/middleware`, `pkg/uos/httpx` share the `pkg/uos` tag (they're in the same Go module). To bump only one subpackage, you bump the whole `pkg/uos` module.

`pkg/testkit/contract` is in the same root module as `pkg/uos` in v0.1. (Architecture_plan Â§3.2 places it inside the root module.) It therefore shares the `pkg/uos` tag in v0.1. **Open question for Architect**: if `pkg/testkit/contract` evolves faster than `pkg/uos`, should it be hoisted to its own module? Plan keeps it inside the root for v0.1 simplicity; revisit at M6.

`providers/<name>` modules version independently: `providers/aws/v0.X.Y`, `providers/minio/v0.X.Y`, etc. A `providers/<name>` module declares its compatibility with a `pkg/uos` minor range in its `go.mod` `require`.

### 6.2 The "â‰Ą2 provider justification" rule

Adding any new `Code` or `Capability` requires:

1. **Two independent providers**, in *different* shipped milestones, demonstrably needing the same semantic. "Independent" means not in the same vendor family (S3-family providers count as one; AWS + Azure count as two; AWS + Alibaba count as two â€” Alibaba OSS is its own SDK and its own auth dialect; OSS + Tencent count as two; AWS + MinIO is **not** two â€” both are S3 reference).
2. **A documented matrix update**: `docs/provider_matrix.md` shows the new code/capability with the cells filled in for at least the two justifying providers.
3. **A linked GitHub issue** in this repo containing: the proposal, the two-provider use cases, and the alternatives considered (especially: "could `As(target)` and `Cause` already serve this need?").
4. **A `pkg/uos` minor bump**, with the new constant added to `AllCodes()` / `All()` in declaration order (append, never reorder). The matching surface_test update is part of the same PR.
5. **CHANGELOG entry** on the minor bump explicitly listing the new constant and the two providers.

If only one provider needs the semantic, the answer is `As(target any) bool` (the escape hatch) or `Error.Cause` (the carry-through), not a new constant.

### 6.3 Process: breaking vs. additive vs. internal-only

| Change shape | Examples | Process |
| --- | --- | --- |
| **Breaking** | renaming `Client.Capabilities` to `Client.Caps`; removing `ErrLengthRequired`; adding a method to `ObjectService` | Open RFC issue â†’ 2-week comment window â†’ Architect + Critic review â†’ minor bump if the breakage is deferrable to v2 (use deprecation), or major bump otherwise. Deprecation deferral is preferred when feasible. |
| **Additive (constant)** | adding `ErrConcurrentModification` because Azure ETag and OSS ETag both need it | Follow Â§6.2 in full. |
| **Additive (function/method on non-frozen interface)** | adding a helper `SignURLForRange(req, byteRange)` to `Signer` | Open issue â†’ Architect review â†’ minor bump. (Recall: `Signer` is one of the 5 frozen interfaces â€” additive method requires major. So this specific example is actually a major change. Fix instead by adding a new helper function in `pkg/uos` that wraps `Signer.SignURL`.) |
| **Additive (struct field)** | adding `PutObjectRequest.IfNoneMatch *string` | Issue â†’ minor bump. Field MUST be appended at the end and use a pointer or zero-value-meaningful type to preserve composite-literal compatibility. |
| **Internal-only** | refactoring `transfer.Manager` worker pool; updating a test fixture; adding logging inside an existing function | Patch bump or no bump if no release. No process. |

### 6.4 Rollback / yank policy

- **Critical bug discovered post-tag**: cut a `v1.X.Y+1` patch within 24 hours if mitigation is straightforward; otherwise document the workaround in a pinned GitHub issue and add a deprecation notice to the affected version's release notes.
- **Yanking via `go mod`**: use `retract` directives in `go.mod` rather than deleting tags. Format: `retract v1.X.Y // <reason>`.
- **Major-bump rollback**: a `v2.0.0` that turns out to be wrong is rolled back via `retract v2.0.0` and a `v2.0.1` that re-asserts the v1 surface plus the original v2 fix done differently. We do not delete `v2.0.0` from the tag history.

### 6.5 How M2 will validate that v0.1 didn't lock in the wrong abstraction

This is the post-v0.1 review gate and the most important part of Â§6.

M2's exit checklist (`provider_roadmap.md` M2) explicitly includes: "**No new `Code` or `Capability` was added to `pkg/uos`.**" If M2 needed to add one, that's a v0.1 abstraction defect we caught in time.

The M2 PR template (P5 follow-up, ideally) requires an "Abstraction lessons" section answering:

1. Did `transfer.Manager` orchestrate AWS multipart correctly? If not, what was the mismatch?
2. Did `StateStore`'s opaque `[]byte` payload work for AWS multipart resume, or did the driver want a typed schema?
3. Did `Signer.SignURL` cover AWS presign? Did `SignedURL` carry everything the caller needs?
4. Did `error_map.go` map every AWS error class to one of the 14 codes without ambiguity?
5. Did `capability.Report` cleanly distinguish "AWS supports X with this credential" from "AWS supports X but this bucket isn't configured for it"?
6. Did `As(target any) bool` work for AWS Object Lock and the other in-scope-via-extension features?

If any answer is "no" or "kind of," the corresponding M1 code is changed in `pkg/uos` BEFORE M2 tags. The `provider_roadmap.md` "Lessons" section gets the writeup. If the change is breaking, M2 cuts `pkg/uos/v0.2.0` and M2's drivers tag `providers/aws/v0.1.0` against `v0.2.0`. There is no `v0.1.x` patch series that diverges from `v0.2.x` â€” we move forward, never sideways.

The same protocol applies to M3, M4, M5. M6 cuts `pkg/uos/v1.0.0` only if the surface has been stable for the entirety of M5 (no minor bumps for new codes/capabilities since the last M3-or-later milestone tag).
