# Architect Review — v0.1 Implementation Plan (Round 1)

> **Reviewer**: Architect (deliberate mode, ralplan consensus loop)
> **Date**: 2026-04-27
> **Plan under review**: `.omc/plans/v0.1-implementation-plan.md` (992 lines)
> **Plan recommendation**: Option B — "Skeleton + working defaults"
> **Binding spec**: `docs/architecture_plan.md` (frozen 2026-04-27)

## Verdict: SOUND_WITH_NOTES

The plan is structurally clean and faithfully implements the architecture freeze. The 14-code / 13-capability surface is correctly pinned, the multi-module dependency direction is enforced, and the test strategy is well thought through. Verdict is **not** `SOUND` because pressure-test (c) — the `transfer` package's local adapter types — is a real layering smell that the plan should resolve in v0.1 rather than carry forward, and pressure-test (d) — the `transfer.Manager` shape — has a partial truth in the antithesis that warrants a small, scope-preserving v0.1 hedge. Neither rises to `RECOMMEND_REVISION` because both are addressable with surgical edits to the existing P2 deliverable; no rewrite of Option B is required.

---

## 1. Strengths

1. **Frozen-set surface test as a first-class artifact (P4 / §"Surface" acceptance criteria, plan lines 124-130, 919-924).** The plan correctly identifies that `surface_test.go` carrying a literal duplicate of the 14 codes and 13 capabilities IS the test (the redundancy is the assertion). This is the single highest-leverage anti-erosion mechanism for the architecture's §7 freeze, and the plan implements it bluntly and visibly.

2. **Pre-mortem #2 mitigation is concrete and enforceable (plan lines 65-72).** The plan doesn't just say "freeze the set"; it pairs the surface test with §6.2's `≥2 independent providers in different shipped milestones, in different vendor families` rule and the explicit "AWS + MinIO is NOT two" carveout. That's the precise shape of the rule needed to stop the well-meaning-PR drift that erodes most multi-vendor SDKs.

3. **Module-path placeholder strategy via `go mod edit -module` instead of `sed` (plan line 82, R7).** Architecturally tiny but operationally critical: the plan recognizes that placeholder substitution is a release-blocker risk and routes it through the only Go-native command that won't leave straggler import paths. Most plans in this category miss this detail entirely.

4. **Build tag isolation for testcontainers (plan line 79, P3 / P4).** `//go:build docker` on `pkg/testkit/contract/minio.go` lets the testkit compile cleanly without Docker, separating the contract suite's compile-time contract from its runtime gate. This decouples the M1 exit criterion from environmental friction and preserves architecture_plan §6.2's "contributors without Docker can still build and run unit tests" promise.

5. **Per-module versioning protocol with explicit `retract` policy (plan §6.4, lines 880-884).** The plan correctly lands on `retract` rather than tag deletion for yanks, which preserves git history and respects Go module proxy semantics. Combined with §6.1's "5 frozen interfaces require major bump for any method addition" and §6.5's M2 abstraction-validation gate, the change-control rules form a tight, defensible v1.x discipline.

## 2. Steelman antithesis

### The opposite position

> **Option A is right; Option B is overreach. Ship a pure-interface v0.1, defer all working defaults to M2.**

The plan's case for Option B rests on two claims: (1) "M2 driver author is unblocked on day 1," and (2) "abstraction defects in `transfer.Manager` surface in M1 against unit tests rather than in M2 against a real driver." Both claims are weaker on close inspection than the plan presents.

Claim (1) is misleading because Option B does not actually unblock M2's hardest work — the AWS multipart wiring, the SigV4 dance, and the AWS error → 14-code mapping. Those are M2-only by definition. What Option B unblocks is the *easy* M2 work (registry lookup, middleware composition, HTTP client construction). Optimizing v0.1 to remove easy M2 work, at the cost of locking in an unvalidated `transfer.Manager` shape, is a poor velocity trade.

Claim (2) is the load-bearing argument and it is the one the antithesis must dismantle. **The premise — that unit tests against a fake `MultipartServiceLike` validate the abstraction — is false.** A unit test against a fake validates the *plumbing*, not the *abstraction*. The abstraction is the bridge between `transfer.Manager`'s coordination model and a real provider's resumable-upload primitives. AWS S3's multipart model is so close to the abstraction's mental model (uploadID + numbered parts + per-part etag) that an AWS-only unit-test fake will look exactly like the abstraction itself — the test is a tautology. The defect that matters is "does this shape fit GCS's session-URL model and Upyun's FORM-direct-upload model," and that defect *cannot* surface in M1 unit tests no matter how thorough they are.

### Why Option A might actually be right

Three reasons:

a. **The cost of an Option-A `pkg/uos/v0.1.0` is genuinely lower than Option B.** Option A is roughly 1,200-1,500 LOC of types + signatures + frozen-set tests. Option B is the plan's own ~3,000-3,500 LOC budget — more than 2× the surface area to review for v0.1. Reviewer attention on a v0.1 should concentrate on *what is frozen forever*, not on *what is a working default that will be rewritten by M4 anyway*. Option A focuses reviewer attention exactly where it matters.

b. **`transfer.Manager` as a v0.1 concrete struct creates a one-way ratchet.** Once `transfer.Manager` ships as a struct (not an interface) with a `(*Manager).Upload(...)` method, every additive method on it — `Download`, `Resume`, `Abort` — has to live with the v0.1 method-receiver shape. If M4 reveals that the right shape is `transfer.UploadStrategy` (an interface chosen by the driver) plus `transfer.Coordinator` (a thin orchestrator), the M1 `*Manager` either gets deprecated (a documented breakage at v0.2) or left as a frozen-but-suboptimal API forever. Option A defers that commitment.

c. **The "M2 unblock" benefit is achievable with a thinner Option A+ε.** Specifically: ship Option A plus *only* `httpx.NewClient`, `credential.StaticProvider`, `credential.Chain`, and the default `Registry`. Skip `transfer.Manager`. M2's AWS author writes the multipart orchestration directly using the AWS SDK's native multipart, then `pkg/uos/transfer.Manager` is designed in M3 with two providers' real-world pressure (AWS + Alibaba) on the abstraction simultaneously. **This is strictly better evidence for the abstraction than Option B's M1-against-fake validation.**

### What would have to be true for Option B to be wrong

Option B is wrong if any of the following holds:

i. **The `transfer.Manager` shape committed in M1 needs a breaking change in M3-M5.** Probability: per the plan's own pre-mortem #1, this is "Medium." That is not a low number for a thing the plan freezes in v0.1.

ii. **The M1 unit tests for `transfer.Manager` give false confidence.** The plan's §4.1 row for `pkg/uos/transfer` says "Manager tests use a fake `MultipartServiceLike` defined in `manager_test.go`." A fake that the same author writes in the same week to make `transfer.Manager` pass cannot validate the abstraction-vs-provider boundary. This is the single weakest spot in the test plan.

iii. **The AWS author in M2 ends up *not* using `transfer.Manager`.** AWS SDK v2 has a high-quality `manager.Uploader` already. If `providers/aws` delegates multipart to `manager.Uploader` (recommended by `provider_roadmap.md` M2 — "Multipart: delegate to S3 native multipart"), then `pkg/uos/transfer.Manager` is *unused* for AWS, and the only M2 driver that exercises it is `providers/minio` (which also has minio-go's native multipart). **Option B may ship a `transfer.Manager` that no M2 driver actually invokes**, in which case the abstraction-defect-detection benefit Option B claims over Option A is zero.

Item (iii) is the steelman's strongest move. It is concrete, citable from `provider_roadmap.md` lines 38-39 and 49-50, and it directly contradicts the plan's recommendation reasoning.

## 3. Tradeoff tensions

### Tension 1: "Validate abstraction in M1" vs. "Don't ship code no driver uses"

The plan picks "validate in M1" by writing `transfer.Manager` against a unit-test fake. The cost is that **M2 may not exercise it** (per `provider_roadmap.md` M2 explicitly delegating multipart to native SDKs), which means the v0.1 `transfer.Manager` ships into v1.0 having only ever been validated against fakes. **Quantified cost**: ~250 LOC of `manager.go` + ~150 LOC of `manager_test.go` (per plan §2 LOC budget) = ~400 LOC of v0.1 surface that may be locked into v1.x without ever having seen a real provider. If M3 (Alibaba) is also delegated to vendor SDK, the first real-provider exercise of `transfer.Manager` is M4 (GCS) — 6+ months after v0.1 tag, against a *non-multipart* upload model the abstraction wasn't shaped for. That's the worst-case scenario for an architectural commitment: long latency to validation + first validation against the antithetical use case.

### Tension 2: "Frozen interface stability" vs. "M2 abstraction-validation freedom"

Plan §6.1 declares the 5 service interfaces (`Client`, `BucketService`, `ObjectService`, `MultipartService`, `Signer`) "append-only requires major bump." Plan §3.1 then adds `ObjectService.Exists` and `MultipartService.List` *beyond* PRD §4.3's baseline, justifying via "PRD FR-005 names it" and "operationally necessary for orphan cleanup." Both additions are reasonable, but the plan picks them now, **at the moment of freezing**, when M2 hasn't validated whether the abstraction even has the right shape. **Cost**: if M2 reveals that, say, `Exists(req HeadObjectRequest)` is the wrong signature (some providers want a richer probe; some want to fold it into `Head` returning a typed `NotFoundError`), the change is a `pkg/uos/v2.0.0`. The plan's own §6.5 explicitly forbids the v0.1.x → v0.2.x patch series ("we move forward, never sideways"), which means a wrong `Exists` signature cascades to every shipped provider.

The tension is real: pre-freezing methods that the architecture freeze deliberately *deferred to PRD* (per architecture_plan §1.2: "Sub-service signatures follow PRD §4.3 verbatim") trades a small day-1 ergonomic win for a non-trivial v2.0.0 risk.

### Tension 3: "Strict NFR-008 (no transitive cloud SDK pollution)" vs. "Test ergonomics"

The plan correctly enforces NFR-008 at the module level via the multi-module layout (architecture_plan §3, plan §"Module structure" acceptance criteria). But `pkg/testkit/contract/minio.go` imports `testcontainers-go`, which transitively pulls Docker SDK + a non-trivial dependency tree. Any provider that wires the contract suite into its own `_test.go` *also* pulls that tree at test-time. **Cost**: `providers/aws` (and every later provider) ships a `go.sum` whose test-only deps include ~30-50 transitive packages from `testcontainers-go`. That's not a runtime cost (test deps aren't in the production binary), but it IS a cost on `go mod download` time, on dependabot/renovate noise, and on supply-chain attack surface for the test environment. The plan accepts this without naming it. It's the correct trade — you can't run real S3-compat contract tests without Docker — but a SOUND_WITH_NOTES verdict requires flagging that NFR-008 is honored *only at runtime*, not at test-time, and that the testkit module-hoist question (plan §6.1 open question) becomes more pressing in v1.x for exactly this reason.

## 4. Synthesis

The steelman exposed two real partial truths that can be absorbed into Option B without rewriting it:

**Synthesis A — De-risk `transfer.Manager` to a v0.1 minimum that no driver is forced to use.**

Keep the Option B recommendation, BUT shrink `transfer.Manager` in v0.1 to:

- The `Config`, `UnknownSizePolicy`, `StateStore` (interface + `MemoryStateStore`), and `RateLimiter` types — these ARE part of the public surface and need to be frozen.
- A `Manager` struct that wraps `Config` and exposes only `Upload(ctx, req UploadRequest) (*UploadResult, error)` where `UploadRequest` is a *concrete struct* (not an adapter interface) carrying the bytes / size / metadata + a `MultipartUploader` interface field that the *driver* fills in.
- **Defer the actual orchestration body to M2.** In M1 the body is `panic("transfer.Manager.Upload: not implemented in M1; tracked in providers/aws M2")` plus a one-line comment pointing at the M2 work item.

This preserves Option B's "frozen public Manager type ships in v0.1" benefit while eliminating Option B's "ship 400 LOC of orchestration validated only against fakes" cost. The driver author in M2 implements the body and writes the *real* tests against AWS's actual multipart, *then* the plan's §6.5 abstraction-validation gate fires on real evidence. If this body needs to grow strategy variants (multipart vs. session vs. FORM) in M4/M5, those grow as additional unexported helpers behind the same public `Upload` entrypoint — additive, non-breaking.

This synthesis costs the plan one paragraph: in §3.6, change "Manager skeleton: planner, worker pool, abort-on-failure cleanup" (line 110, P2 deliverable) to "Manager type + Config + StateStore (interface + memory impl). Manager.Upload body is a panic stub deferred to M2."

**Synthesis B — Add `Exists` and `MultipartService.List` AFTER M2 confirms their signatures.**

Drop both additions from v0.1. Re-add them in `pkg/uos/v0.2.0` as part of M2's "abstraction-validation lessons" loop. The plan's §6.5 already provides the protocol; using it for these two methods costs one minor bump on `pkg/uos` between M1 tag and M2 tag, which the plan explicitly allows ("If M2 needed to add one, that's a v0.1 abstraction defect we caught in time" — §6.5 last paragraph).

This synthesis is *optional* — it depends on whether the maintainer prefers "v0.1 ships the maximally-complete public surface and risks one method's signature being wrong" (plan's current pick) or "v0.1 ships only the architecture-frozen surface and lets the first real driver pin the rest" (synthesis pick). Both are defensible. The Critic should weigh this against the surface-stability principle.

## 5. Principle violations

### Principle: Multi-module dependency direction (architecture_plan §3.3)

**Plan section**: §3.6 import direction note (lines 612-613).

**Verdict**: **borderline**.

Architecture_plan §3.3 declares `pkg/testkit/contract MAY depend on pkg/uos and on testcontainers-go / Docker SDK; nothing else.` It does NOT explicitly forbid `pkg/uos/transfer` (a subpackage) from defining adapter types — but the plan's own admission that the adapters exist *to dodge an import cycle* (line 613) is the architectural smell. The right reading of architecture_plan §3.2's directory layout is that `pkg/uos/transfer` is a *child* of `pkg/uos` and child packages cannot import their parent in Go. The fix the plan picks (option `(a)` — local adapter types in `transfer`) honors the package boundary at the cost of duplicating type information across two packages, which is a maintenance burden that grows monotonically as `transfer.Manager` grows.

The architectural principle being bent is not "dependency direction" itself — it's the principle that **packages should not be created where the dependency direction is awkward**. The cleaner fix (the plan's option `(b)` — move `Manager` to `pkg/uos/transfer.go` flat, or even up to `pkg/uos.Manager`) is what architecture_plan §3.2's table actually permits if you read it strictly: §3.2 lists `transfer.Manager` as a top-level entity; it does NOT mandate that `Manager` lives in a subpackage. The subpackage choice was made for organizational tidiness in §3.2's directory tree, not as a binding constraint.

**Recommendation**: Not a hard violation, but worth flagging to the Critic. The plan should either (i) commit to option `(b)` (move `Manager` up; subpackage `transfer` keeps `Config` / `StateStore` / policies; `Manager` lives in `pkg/uos/manager.go` and freely uses `pkg/uos.PutObjectRequest`), OR (ii) explicitly justify option `(a)` with a doc comment in `transfer/manager.go` explaining the adapter-types pattern. The current plan picks `(a)` without resolving the awkwardness.

### Principle: Frozen surface (architecture_plan §1, §7)

**Plan section**: §3.1 (added `ObjectService.Exists`, `MultipartService.List`); §3.9 (added typed `DirectGrantMode`).

**Verdict**: **no violation, but borderline on `Exists` / `List`**.

Architecture_plan §1.2 says "Sub-service signatures follow PRD §4.3 verbatim. Where this freeze does not override a PRD field, take PRD as authoritative for that field's name and type." PRD §4.3 lists `Exists` only in the FR-005 narrative (PRD line 100), not in the typed interface signatures. The plan's `ObjectService.Exists(ctx, HeadObjectRequest) (bool, error)` is *interpolated* from FR-005 and from architecture_plan §6.4's contract case enumeration ("Object: ... / Exists / ...", architecture_plan line 300). This interpolation is reasonable but is technically an extension beyond the literal text of §1.2.

`MultipartService.List` is similarly interpolated, justified by `provider_roadmap.md` cross-cutting risk #5 ("Multipart orphan"). Reasonable, but again interpolated.

`DirectGrantMode` typed constants extend PRD §4.4's `Mode string`. Per architecture_plan §1's stability ontology, `DirectGrant` is a frozen entity but its *fields* are not individually frozen at the type level. Typing `Mode` to a 4-value enum is a forward-compatible move *if* the 4 values are correct; it is a v2.0.0-class breakage if a 5th value is needed and PRs add it as a constant rather than as a runtime-validated string.

**Recommendation**: The plan's three deviations are within the spirit of the architecture freeze but should be either (a) absorbed into a brief addendum to architecture_plan §1.2, OR (b) explicitly demoted to "plan picks; revisit in M2" status. Currently the plan documents them under §"Deviations from architecture_plan" (line 982) but presents them as final rather than tentative.

### Principle: NFR-008 / no-vendor-SDK-in-core (architecture_plan §3.3)

**Plan section**: §"Module structure" acceptance criteria (lines 930-931).

**Verdict**: **no violation**.

The plan's grep-based assertion (`grep -E '(aws-sdk-go|minio-go|aliyun|...)' go.sum returning empty`) is a strong, automatable enforcement of NFR-008 at the root module level. This honors architecture_plan §3.3's "pkg/uos MUST NOT import any third-party cloud SDK" rule. The plan even goes one step further with the `go list -deps ./pkg/uos/... | grep -v ... | sort -u | <stdlib only>` check in the same acceptance block. Both are correct and tight.

### Principle: Capability frozen at 13 (architecture_plan §7.2)

**Plan section**: §3.4 (capability constants).

**Verdict**: **no violation, exemplary execution**.

The plan declares all 13 in the documented order, exposes `All()` returning them in declaration order, and pins them in `surface_test.go`. The `Report.Items map[Capability]CapabilityStatus` shape with the "drivers MUST fill every key" enforcement rule (line 487) is exactly what the architecture's capability-completeness contract requires.

## 6. Targeted pressure-test of Planner's 4 flagged decisions

### (a) `ObjectService.Exists` and `MultipartService.List`

**Verdict on `Exists`**: **Justified, neutral risk.**

PRD FR-005 (`docs/prd.md:100`) literally lists `Exists` as a P0 functional requirement. Architecture_plan §6.4 (line 300) lists it in the contract case enumeration. The plan's signature `Exists(ctx, HeadObjectRequest) (bool, error)` is the right shape because it lets drivers reuse `Head`'s wire path and decode at the boundary. The alternative — forcing every caller to `Head` and pattern-match `errors.Is(err, ErrNotFound)` — is operationally annoying and is a classic source of bugs (callers conflate "not found" with "permission denied"). Adding it now costs nothing the architecture has not already implicitly committed to.

**Verdict on `MultipartService.List`**: **Justified, slight-positive risk.**

`ListMultipartUploadsRequest` is needed by the orphan-cleanup scenario in `provider_roadmap.md:179`. Without it, a user's "list and abort orphan uploads" operational task forces them out to `As(target)`, which defeats the purpose of the unified Multipart abstraction. All S3-family SDKs expose this; GCS exposes it via a different shape (resumable session URLs are listed differently); Azure exposes it via blob list with uncommitted-blocks filter. The signature `List(ctx, ListMultipartUploadsRequest) (*MultipartUploadList, error)` is the S3 shape, which means GCS and Azure drivers will translate from their native shapes — consistent with the rest of the abstraction's design philosophy.

**One nit**: the plan should add the literal `MultipartUpload` and `MultipartUploadList` struct definitions to §3 right alongside `MultipartService.List`, or it leaves the v0.1 reviewer guessing what those structs hold. This is a documentation gap, not an architecture concern.

**Synthesis-aware alternative**: per Synthesis B above, defer both methods to `pkg/uos/v0.2.0` (M2). The maintainer's call. If the plan keeps them in v0.1, no harm done, but the surface is one method-signature wider than strictly required.

### (b) Typed `DirectGrantMode` (4 constants)

**Verdict**: **Acceptable risk; recommend keeping as typed enum.**

The 4 modes (`url`, `form`, `token`, `headers`) cover the documented vendor matrix: Azure SAS = `url` or `headers`; Upyun FORM = `form`; Qiniu Upload Token = `token`; per-request signed headers = `headers`. None of the 10 in-scope providers needs a 5th mode that isn't representable as one of these four. The typed enum is *strictly safer* than `Mode string` because:

- It forces drivers into the documented dispatch space at compile time.
- `surface_test.go` can pin the 4 values, making accidental drift loud (the plan already commits to this in §3.9).
- `errors.As`-style pattern matching on `DirectGrant.Mode == GrantModeForm` becomes valid at compile time, not at runtime.

**However**, if a future provider needs a 5th mode (hypothetical: a "post-with-multipart-form" mode that doesn't fit Upyun's FORM shape), the addition is a `pkg/uos` minor bump under §6.2's ≥2-provider rule. The plan's R9 risk row already accepts this. The mitigation is correct: ship the 4-value enum, pin it in `surface_test.go`, document the addition protocol.

**Critique of the alternative**: Keeping `Mode string` shifts the validation burden onto runtime ("did the driver return a mode the caller knows how to dispatch on?") and onto every caller's switch statement having a `default:` arm. That's strictly worse for type-safety in a frozen v1 API.

**Recommendation**: keep the typed `DirectGrantMode`. Rename the deviation flag from "minor" to "intentional design choice, frozen-by-≥2-provider-rule for future additions." Update §3.9 accordingly.

### (c) `transfer` package using local adapter types

**Verdict**: **The cycle is real, but the layering choice is debatable. Recommend option `(b)`.**

The cycle is real: `pkg/uos.Client.Multipart(bucket).Upload(...)` wants to delegate to `transfer.Manager.Upload(...)`, which needs `pkg/uos.PutObjectRequest` to know what to upload. If `transfer` imports `pkg/uos`, that's a cycle (pkg/uos already imports `pkg/uos/transfer` for `Config`).

The plan's option `(a)` (local adapter types `UploadRequest`, `MultipartServiceLike`, etc. in `transfer`) is a valid Go pattern but it has a real cost: **every field added to `pkg/uos.PutObjectRequest` in v1.x must be mirrored in `transfer.UploadRequest`, with a translation function in `pkg/uos`**. That is a maintenance tax that grows monotonically as the public surface evolves.

The plan's option `(b)` (move `Manager` up, e.g., to `pkg/uos/manager.go` directly) eliminates the cycle by collapsing the layer. The `transfer` subpackage retains `Config`, `UnknownSizePolicy`, `StateStore`, `RateLimiter` — which are pure value types with no dependency on `pkg/uos` types. `Manager` itself becomes `pkg/uos.Manager` (or `pkg/uos.Uploader`) and freely uses `PutObjectRequest`, `MultipartService`, `ObjectService`. **This is structurally simpler, eliminates the adapter-type maintenance tax, and matches PRD §4.11's narrative more closely (the PRD does not actually nest Manager in a subpackage — it discusses it inline with the rest of the unified API).**

The architecture_plan §3.2 directory tree lists `transfer/manager.go` as a path, but architecture_plan §1's entity table lists `transfer.Manager` as the public name. There is room to read §3.2's tree as illustrative (the layout the planner *could* pick) and §1's name as binding (the *exported name* must be `transfer.Manager`). Under that reading, you can satisfy both by placing the file at `pkg/uos/transfer/manager.go` *and* keeping the import direction `transfer → pkg/uos`, IF you split `pkg/uos` into a deeper layer (e.g., `pkg/uos/internal/types` holding the request structs) that both can import. That's a third option — `(c)` — and it's strictly more complex than `(b)`.

**Will the adapters leak abstraction in M4?** Yes, with high probability. Once GCS arrives, `transfer.Manager` will need to know about a session URL (not a uploadID + parts). The adapter `MultipartServiceLike` will either grow a `SessionLike` cousin (now you have N adapter shapes for M shapes), or `transfer.UploadRequest` will grow a `SessionURL string` field that is meaningless for AWS but populated by GCS — both of which leak abstraction across the package boundary the adapters were supposed to protect.

**Recommendation**: Switch to option `(b)` in v0.1. Move `Manager` (or rename to `Uploader`) to `pkg/uos/uploader.go`. The `transfer` subpackage holds only the value types (`Config`, `UnknownSizePolicy`, `StateStore` interface, `MemoryStateStore`, `RateLimiter`). The cycle disappears, the adapter-type maintenance tax disappears, and the layering is cleaner. **This is a P2 deliverable change; it does not expand v0.1 scope; it slightly shrinks it.**

### (d) `transfer.Manager` design under M4 GCS resumable uploads

**Verdict**: **The bias risk is real but the plan's mitigation is mostly correct. One additional v0.1-time hedge is recommended.**

Let me unpack this carefully because it's the one the planner flagged as highest-bus-factor.

**Does the design lock in S3-multipart bias?** The plan's `Manager.Upload(ctx, mp MultipartServiceLike, obj ObjectServiceLike, req UploadRequest) (*UploadResult, error)` has an `mp MultipartServiceLike` parameter, which IS the S3-multipart bias. GCS resumable uploads do not have an `Initiate / UploadPart / Complete / Abort` lifecycle; they have a `BeginSession → PUT-with-Range → finalize` lifecycle. The `MultipartServiceLike` interface shape forces GCS's driver to either (i) wrap its session API in a fake multipart facade (unnatural, error-prone), or (ii) bypass `transfer.Manager` entirely for GCS uploads (which, per the steelman in §2.iii, may happen anyway).

**Is `StateStore` opaque `[]byte` enough?** Yes. The plan's mitigation here is correct (per pre-mortem #1, plan lines 56-63). `[]byte` is the right escape — drivers serialize their own resume schema, and `transfer.Manager` is a *coordinator* that routes the bytes between provider and disk without inspecting them. This decouples the StateStore from the upload model.

**But the issue is upstream of `StateStore`.** `StateStore` is fine; the problem is that the *coordinator* (`Manager.Upload`) hardcodes the multipart lifecycle in its planner. If the planner picks "single-shot Put" or "multipart upload," there is no third arm for "session-style upload." The choice between single-shot and multipart is hardcoded against `mp MultipartServiceLike`.

**Will this force `pkg/uos/v2.0.0` within 8 weeks of M4?** Possibly, but probably not directly because of `transfer.Manager`. More likely:

- M4 GCS will simply *bypass* `transfer.Manager` (driver implements its own resumable-upload orchestration). Then `transfer.Manager` is "S3-only" by convention, documented as such, and the v1.x abstraction holds — at the cost of being a half-abstraction.
- If the maintainer insists `transfer.Manager` MUST cover GCS, then yes — a v2.0.0 of `pkg/uos` is needed, because the `Upload(ctx, mp MultipartServiceLike, obj, req)` signature cannot accommodate a third primitive without a method addition.

**The v0.1-time mitigation that doesn't expand scope**:

Refactor `Manager.Upload`'s second parameter from a typed `mp MultipartServiceLike` to an interface that *abstracts the upload primitive*, not the multipart shape:

```
type Uploader interface {
    // Upload performs the entire upload, returning the result. The Manager owns
    // partitioning, retry policy translation, progress callback wiring, and
    // StateStore plumbing; the Uploader owns the wire protocol.
    Upload(ctx context.Context, plan UploadPlan, store StateStore) (*UploadResult, error)
}

type UploadPlan struct {
    Body       io.Reader
    Size       int64  // -1 if unknown
    PartSize   int64  // 0 if Uploader chooses
    Concurrency int
    Resume     []byte  // opaque payload from previous attempt; nil on first try
    Progress   ProgressCallback
}
```

Under this shape:

- AWS / MinIO drivers wrap their multipart code into an `Uploader` implementation that translates `UploadPlan` into `Initiate / UploadPart×N / Complete`.
- GCS driver wraps its resumable-session code into an `Uploader` that translates `UploadPlan` into `BeginSession / PUT-with-Range×N / finalize`.
- Upyun driver wraps FORM upload as a single-shot `Uploader`.
- `transfer.Manager` owns: planner (do we need `Uploader` at all, or can we single-shot Put via `ObjectService`?), retry, progress, StateStore plumbing, abort-on-failure protocol.
- `transfer.Manager` does NOT own: the wire protocol's shape.

**This shift moves the abstraction boundary one notch up**. It costs ~80 LOC of v0.1 work (define `Uploader` interface, define `UploadPlan` struct, simplify `Manager.Upload` signature). It is pure additive design work; nothing in the plan's other deliverables changes.

**This is the recommended v0.1 hedge for pressure-test (d).** Combined with Synthesis A (Manager body deferred to M2 via `panic`), the v0.1 commitment becomes:

- Public types (`Manager`, `Config`, `Uploader`, `UploadPlan`, `StateStore`, `MemoryStateStore`, `UnknownSizePolicy`) — frozen.
- `Manager.Upload` body — `panic("not implemented in M1; tracked in providers/aws M2")`.
- M2 author writes the body against AWS multipart, validates the abstraction against a real driver.
- M4 GCS author writes a session-style `Uploader`; the planner accommodates without changing.

If you accept this hedge, the answer to the planner's specific question — "does this design lock in an S3-multipart bias that will require `pkg/uos/v2.0.0` within 8 weeks of M4?" — becomes "no, because `Uploader` is the explicit extension point and GCS plugs into it without touching `pkg/uos`."

If you don't accept this hedge, the answer is "**yes, with high probability**, M4 forces either a `v2.0.0` bump or a 'GCS bypasses transfer.Manager' carve-out. The planner's framing is correct: this is the highest-bus-factor decision in v0.1."

## 8. Items that the Planner got wrong but you would NOT block on

These are noted for v0.1.x or v1.0 polish; they should not gate v0.1 sign-off:

1. **`EnvProvider` reads both `AWS_*` and `OSC_*` vars (plan line 110, P2 deliverable (c)).** Reading `AWS_ACCESS_KEY_ID` in `pkg/uos/credential` couples a vendor-neutral provider to a specific vendor's env-var convention. This is a small vendor-leak. Better: have `EnvProvider` only read `OSC_*` (the project's own convention) and let `providers/aws/credential.go` extend with `AWSEnvProvider` for AWS env vars. Defer to v0.2.0; not blocking.

2. **`RetryPolicy.RetryableCodes []Code` defaults documented in a comment (plan line 339).** The default ("typically: rate_limited, timeout, temporary") is normative — it will be relied on by every driver. It should be a named exported variable: `var DefaultRetryableCodes = []Code{ErrRateLimited, ErrTimeout, ErrTemporary}`. Lets callers append rather than redeclare. Defer to v0.2.0.

3. **`Error.Retryable bool` is redundant with `RetryableCodes`** (plan line 416). If retry-decision is "is `Code` in `RetryableCodes`," the `Retryable bool` field on `*Error` is either redundant (when the driver computes it from Code) or contradictory (when the driver sets it independently). Recommend: drop `Retryable bool` from `*Error`, derive at the retry layer. But this is a frozen-surface concern — if it's in v0.1 it stays. Note for v0.2.0 review.

4. **`go.work` policy says it's "committed" (architecture_plan §3.6) but the plan doesn't address `vendor/` interaction.** Some downstream consumers vendor their dependencies; `go.work` is intentionally ignored under `-mod=vendor`. Worth a note in `RELEASING.md`, but not v0.1-blocking.

5. **The `Logger` / `Metrics` / `Tracer` interface trio in middleware (plan §3.7) should arguably be one `Observer` interface with three Observe methods.** The current shape forces three separate nil-checks at every call site; a single composite interface flattens that. Style preference; not blocking.

6. **`AGENTS.md` Appendix A item #1 update is in P5 (plan §"Documentation" acceptance criteria) but the AGENTS.md path it cites is currently broken** (architecture_plan Appendix A.1 says current `requirements_and_design.md` does not exist). This is documented elsewhere; the plan handles it correctly. No action.

7. **The CI `unit` job runs on Windows (plan §4.5).** Testcontainers-go has known issues on Windows with Docker Desktop. The `unit-docker` job is correctly Linux-only, but the plan should explicitly note that contributors developing on Windows can still work locally without `unit-docker`. Doc nit, not blocking.
